# Official Channel Posting Runbook

## Recommended order

1. Publish a GitHub release and public review issue only after explicit human
   approval.
2. Use Cursor Forum, Showcase, or Ideas for public technical feedback.
3. Use Graphite Community Slack or public GitHub/contact route.
4. Use MCP GitHub Discussions.
5. Optionally use `security@cursor.com` only for narrow agent security, audit,
   or replay evidence architecture feedback.

## Cursor / Origin

1. Join the Origin waitlist.
2. Post a technical feedback request in Cursor Forum Ideas, Showcase, or Discussions.
3. Use Contact Sales only if framing as enterprise or research protocol feedback.
4. Do not claim official Origin integration.
5. Do not use `security-reports@cursor.com` unless reporting a real
   vulnerability.
6. Do not use `hi@cursor.com` for technical protocol outreach.

## Graphite

1. Use Community Slack or a public GitHub/contact route.
2. Mention the stacked PR and agent PR evidence problem.
3. Link the release or review issue only after the public release exists.
4. Do not use `security@graphite.dev` or `security@graphite.com` for generic
   protocol outreach.

## MCP

1. Start with GitHub Discussions.
2. Use Issues only for actionable concrete proposals.
3. For protocol-level change, follow the SEP route and seek a sponsor.
4. Mention the MCP bridge as an external evidence and audit boundary, not as an MCP protocol replacement.
5. Do not send email; no verified MCP public outreach email is listed.
